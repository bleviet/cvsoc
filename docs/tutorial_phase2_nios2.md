By the end of this tutorial you will have a **complete embedded system** running inside the FPGA fabric of your DE10-Nano board:

- A **Nios II/e soft-core CPU** executing C firmware

## Concepts in 5 minutes

Before writing a single line, spend 5 minutes with these ideas. They are the mental model you will need throughout the tutorial.

### The Nios II soft-core CPU

A **soft-core CPU** is a processor described in HDL and synthesised into FPGA logic cells — no dedicated silicon required. Intel's Nios II is such a processor. The variant used here, **Nios II/e (Tiny)**, trades performance for a minimal logic footprint: it fits comfortably alongside application logic in a mid-range Cyclone V device.

The CPU itself does not exist as a hand-written VHDL file. It is generated by a tool called **Platform Designer** (formerly Qsys).

### Platform Designer and the Avalon-MM bus

**Platform Designer** is Intel's system integration tool. You describe a set of IP components and their connections in a `.qsys` file; Platform Designer generates all the glue logic (address decoders, bus interconnect, reset sequencing) automatically.

Components communicate over the **Avalon Memory-Mapped (Avalon-MM)** bus. This bus follows a simple master/slave model:

- **Masters** initiate transactions (read or write to an address). In this project, the Nios II CPU has two masters: a data master and an instruction master.
- **Slaves** respond to transactions (they hold registers or memory at a fixed address). The on-chip RAM, PIO, and JTAG UART are all slaves.

Because every peripheral appears at a fixed address in the CPU's address space, controlling a peripheral from C code is just a memory write:

```c
// Write 0xAA to the register at address LED_PIO_BASE
*(volatile uint32_t *)LED_PIO_BASE = 0xAA;
```

The HAL (see below) wraps this into a named macro so you do not need to hard-code addresses.

### The HAL Board Support Package (BSP)

The **HAL (Hardware Abstraction Layer)** is a thin embedded C runtime generated by the Nios II toolchain from your Platform Designer system description. It provides:

- `system.h` — `#define LED_PIO_BASE 0x00010010` and every other peripheral base address
- `altera_avalon_pio_regs.h` — macros like `IOWR_ALTERA_AVALON_PIO_DATA(base, data)`
- A linker script, startup code, and minimal C library

The BSP is generated once per hardware design. When you change the hardware, you regenerate the BSP.

## Step 1 — Design the hardware system

The entire Platform Designer system is defined in a single TCL script: `qsys/nios2_system.tcl`. This script is the authoritative description of every component, every connection, and every base address. No `.qsys` XML file is checked in — it is always regenerated from this script.

### 1.1 Open the script

```
04_nios2_led/qsys/nios2_system.tcl
```

### 1.2 Understand the structure

The script follows a consistent pattern. Read it from top to bottom:

**Boilerplate header** — every qsys-script file must begin with this exact line:

```tcl
package require -exact qsys 12.0
```

**Create the system and target the device:**

```tcl
create_system nios2_system
set_project_property DEVICE_FAMILY {Cyclone V}
set_project_property DEVICE {5CSEBA6U23I7}
```

**Add instances** — each component is added with `add_instance <name> <type>`. The name is yours to choose; the type is the Intel IP component identifier:

```tcl
add_instance clk_0     altera_clock_bridge
add_instance nios2     altera_nios2_gen2
add_instance onchip_mem altera_avalon_onchip_memory2
add_instance jtag_uart altera_avalon_jtag_uart
add_instance sysid     altera_avalon_sysid_qsys
add_instance led_pio   altera_avalon_pio
```

**Configure instances** — set parameters with `set_instance_parameter_value`:

```tcl
# 32 KB on-chip RAM, 32-bit wide
set_instance_parameter_value onchip_mem memorySize {32768}
set_instance_parameter_value onchip_mem dataWidth   {32}

# 8-bit output PIO, starts at 0 (all LEDs off)
set_instance_parameter_value led_pio width     {8}
set_instance_parameter_value led_pio direction {Output}
set_instance_parameter_value led_pio resetValue {0}
```

**Wire clocks** — every component needs a clock. All components share the single 50 MHz clock from the board:

```tcl
add_connection clk_0.out_clk nios2.clk
add_connection clk_0.out_clk onchip_mem.clk1
add_connection clk_0.out_clk jtag_uart.clk
add_connection clk_0.out_clk led_pio.clk
# ... and the others
```

**Wire resets** — resets flow from the reset bridge to every peripheral:

```tcl
add_connection reset_bridge.out_reset nios2.reset
add_connection reset_bridge.out_reset onchip_mem.reset1
# ...
```

**Wire the Avalon-MM bus** — the CPU has two bus masters. Each slave that the CPU can access must be connected to at least one master:

```tcl
# Data master: everything the CPU can read/write at runtime
add_connection nios2.data_master onchip_mem.s1
add_connection nios2.data_master led_pio.s1
add_connection nios2.data_master jtag_uart.avalon_jtag_slave
add_connection nios2.data_master sysid.control_slave
add_connection nios2.data_master nios2.debug_mem_slave   # ← required

# Instruction master: only RAM (the CPU fetches code from here)
add_connection nios2.instruction_master onchip_mem.s1
add_connection nios2.instruction_master nios2.debug_mem_slave  # ← required
```

> **Key learning #1 — Both bus masters must connect to `debug_mem_slave`.**  
> When the Nios II debug port is enabled (the default), `qsys-generate` requires that *both* `data_master` and `instruction_master` are connected to `nios2.debug_mem_slave`. Connecting only the data master will produce a hard error at generation time.

**Assign base addresses** — each slave connection must have a unique, non-overlapping base address:

```tcl
set_connection_parameter_value nios2.data_master/onchip_mem.s1       baseAddress {0x00000000}
set_connection_parameter_value nios2.data_master/led_pio.s1          baseAddress {0x00010010}
set_connection_parameter_value nios2.data_master/jtag_uart.avalon_jtag_slave baseAddress {0x00010100}
set_connection_parameter_value nios2.data_master/sysid.control_slave baseAddress {0x00010108}
set_connection_parameter_value nios2.data_master/nios2.debug_mem_slave baseAddress {0x00010800}
# Instruction master needs its own base address assignments too:
set_connection_parameter_value nios2.instruction_master/onchip_mem.s1 baseAddress {0x00000000}
set_connection_parameter_value nios2.instruction_master/nios2.debug_mem_slave baseAddress {0x00010800}
```

**Wire interrupts** — the JTAG UART requests an interrupt when data arrives. The Nios II IRQ model is slightly counterintuitive:

```tcl
add_connection nios2.irq jtag_uart.irq
set_connection_parameter_value nios2.irq/jtag_uart.irq irqNumber {0}
```

> **Key learning #2 — The IRQ connection direction is counterintuitive.**  
> `nios2.irq` is the *master* (the receiver): the CPU's interrupt controller. `jtag_uart.irq` is the *slave* (the sender). The connection is written as `nios2.irq jtag_uart.irq`, not the other way around. This follows the Platform Designer "interrupt sender/receiver" model, not the electrical signal direction.

**Set reset and exception vectors** — tell the CPU where to begin executing after reset and where to jump on an exception:

```tcl
set_instance_parameter_value nios2 resetSlave      {onchip_mem.s1}
set_instance_parameter_value nios2 resetOffset     {0x00000000}
set_instance_parameter_value nios2 exceptionSlave  {onchip_mem.s1}
set_instance_parameter_value nios2 exceptionOffset {0x00000020}
```

Both vectors point into the on-chip RAM. The reset vector is at offset `0x0` (the very start of RAM); the exception vector is at offset `0x20` (32 bytes in), which is the standard Nios II layout.

**Export top-level interfaces** — three signals must be visible to the surrounding VHDL:

```tcl
add_interface clk clock sink
set_interface_property clk EXPORT_OF clk_0.in_clk

add_interface reset reset sink
set_interface_property reset EXPORT_OF reset_bridge.in_reset

add_interface led_external_connection conduit end
set_interface_property led_external_connection EXPORT_OF led_pio.external_connection
```

**Save the system:**

```tcl
save_system nios2_system.qsys
```

> **Key learning #3 — Run `qsys-script` from the same directory as the output file.**  
> The `qsys-script` TCL interpreter does not implement `file normalize`. If you call `save_system` with a relative path it must be relative to the *working directory where qsys-script is invoked*, not relative to the script file. The Makefile handles this by changing into `../qsys` before running:
> ```makefile
> cd ../qsys && qsys-script --script=nios2_system.tcl
> ```

## Step 2 — Write the VHDL top-level wrapper

The Platform Designer system is an auto-generated black box. Your hand-written VHDL (`hdl/de10_nano_top.vhd`) is the thin wrapper that ties it to the physical pins of the FPGA.

### 2.1 Discover the generated port names

After `qsys-generate` runs, inspect the generated entity to see the exact port names:

```bash
docker run --rm -v /path/to/cvsoc:/work raetro/quartus:23.1 \
  head -20 /work/04_nios2_led/qsys/nios2_system_gen/synthesis/nios2_system.vhd
```

You will see:

```vhdl
entity nios2_system is
  port (
    clk_clk                        : in  std_logic;
    led_external_connection_export : out std_logic_vector(7 downto 0);
    reset_reset                    : in  std_logic
  );
end entity nios2_system;
```

Platform Designer follows a naming convention: `<interface_name>_<signal_name>`. The interface name `clk` and signal `clk` produce `clk_clk`. The exported conduit `led_external_connection` produces `led_external_connection_export`.

### 2.2 Understand the component declaration

When instantiating a Platform Designer system from VHDL, you **must** use a component declaration rather than a direct entity instantiation:

```vhdl
-- ✅ Correct: component declaration
component nios2_system is
  port (
    clk_clk                        : in  std_logic;
    led_external_connection_export : out std_logic_vector(7 downto 0);
    reset_reset                    : in  std_logic
  );
end component nios2_system;
```

> **Key learning #4 — Platform Designer systems live in their own VHDL library, not in `work`.**  
> The `.qip` file included in your Quartus project registers `nios2_system` under the library named `nios2_system`, not under `work`. If you write `entity work.nios2_system`, Quartus analysis will fail with *"design library 'work' does not contain primary unit 'nios2_system'"*. A component declaration works because the elaborator resolves the binding at link time, after all libraries are loaded.

### 2.3 Wire the signals

The top-level connects three signals:

| Top-level signal | System port | Notes |
|---|---|---|
| `fpga_clk1_50` | `clk_clk` | 50 MHz board clock, straight through |
| `power_on_reset` | `reset_reset` | active-high; generated by shared IP |
| `led` | `led_external_connection_export` | 8-bit output to physical pins |

```vhdl
nios2_system_inst : nios2_system
  port map (
    clk_clk                        => fpga_clk1_50,
    reset_reset                    => power_on_reset,
    led_external_connection_export => led
  );
```

The `power_on_reset_generator` is the same shared IP from phases 0 and 1. It holds reset asserted for ~1 ms after power-up, giving the PLL and on-chip RAM time to stabilise before the CPU starts executing.

## Step 3 — Set up the Quartus project

The file `quartus/de10_nano_project.tcl` creates the Quartus project. The critical addition compared to phases 0 and 1 is the `.qip` file reference, which pulls in the entire generated Platform Designer system:

```tcl
# Existing pattern from phases 0–1
set_global_assignment -name VHDL_FILE ../../common/ip/power_on_reset/power_on_reset_generator.vhd
set_global_assignment -name VHDL_FILE ../hdl/de10_nano_top.vhd

# New for phase 2: include the generated Platform Designer system
set_global_assignment -name QIP_FILE ../qsys/nios2_system_gen/synthesis/nios2_system.qip
```

The `.qip` file is a TCL script generated by `qsys-generate`. It registers all sub-IP source files (dozens of VHDL and System Verilog files) and places them in the `nios2_system` library. You never need to list those files individually.

The `de10_nano.sdc` timing constraints file adds one entry compared to the pure-VHDL phases:

```tcl
create_clock -period "50.0 MHz" [get_ports fpga_clk1_50]

# The Nios II debug unit uses the JTAG test clock
create_clock -name {altera_reserved_tck} -period 41.67 [get_ports {altera_reserved_tck}]
set_clock_groups -exclusive -group [get_clocks {altera_reserved_tck}]
```

The JTAG clock must be declared; without it, the Timing Analyzer reports setup violations on the debug path.

## Step 4 — Compile the FPGA bitstream

The Makefile in `quartus/` orchestrates the entire hardware flow. From inside the Docker container, run:

```bash
docker run --rm \
  -v /path/to/cvsoc:/work \
  raetro/quartus:23.1 \
  bash -c "cd /work/04_nios2_led/quartus && make qsys project compile"
```

What happens at each sub-step:

**`make qsys`** runs two tools:

1. `qsys-script --script=nios2_system.tcl` — reads the TCL and writes `nios2_system.qsys`
2. `qsys-generate nios2_system.qsys --synthesis=VHDL` — reads the `.qsys` and generates the full VHDL hierarchy into `nios2_system_gen/`, plus `nios2_system.sopcinfo` (the hardware description file used later for BSP generation)

**`make project`** runs `quartus_sh -t de10_nano_project.tcl`, creating `.qpf` and `.qsf`.

**`make compile`** runs `quartus_sh --flow compile`, executing Analysis & Synthesis → Fitter → Assembler → Timing Analyzer in one pass.

### Expected output

```

Info: Quartus Prime Fitter was successful. 0 errors, 4 warnings
Info: Quartus Prime Assembler was successful. 0 errors, 1 warning
Info: Quartus Prime Timing Analyzer was successful. 0 errors, 2 warnings
```

The warnings (Nios II deprecation notice, minor timing) are expected and safe to ignore. The bitstream is written to `output_files/04_nios2_led.sof`.

## Step 5 — Generate the HAL BSP

The BSP is generated from `nios2_system.sopcinfo` — the machine-readable description of the Platform Designer system. Every time you change the hardware (rename a component, change a base address, add a peripheral), you must regenerate the BSP.

```bash
docker run --rm \
  -v /path/to/cvsoc:/work \
  raetro/quartus:23.1 \
  bash -c "cd /work/04_nios2_led/quartus && make bsp"
```

The Makefile uses `nios2-bsp-create-settings` directly (not the `nios2-bsp` wrapper):

```makefile
nios2-bsp-create-settings \
  --sopc  $(SOPCINFO) \
  --type  hal \
  --settings $(BSP_DIR)/settings.bsp \
  --bsp-dir  $(BSP_DIR) \
  --script   /opt/intelFPGA/nios2eds/sdk2/bin/bsp-set-defaults.tcl \
  --cpu-name nios2
$(MAKE) -C $(BSP_DIR) WINDOWS_EXE=
```

> **Key learning #5 — The `nios2-bsp` wrapper script breaks inside Docker on WSL2 hosts.**  
> The `nios2-bsp` shell script checks `uname -r` for the string "microsoft" to detect Windows Subsystem for Linux. On a WSL2 host, the Docker container inherits the WSL2 kernel version string and the check fires — even though the container is a regular Linux environment. The script then calls `wslpath` (a WSL-specific binary that is not present inside Docker) to convert paths, and the BSP generation fails with `Can't mkdir -p`.  
>
> **Fix:** Call `nios2-bsp-create-settings` directly, bypassing the wrapper. Also pass `WINDOWS_EXE=` when invoking `make` on the generated BSP Makefile, which has the same WSL2 detection and would otherwise call `nios2-elf-gcc.exe` (which does not exist).

After a successful BSP build, the `software/bsp/` directory contains:

```
software/bsp/
├── system.h            ← #define LED_PIO_BASE, JTAG_UART_BASE, etc.
├── linker.x            ← places .text/.data/.bss in on-chip RAM
├── libhal_bsp.a        ← compiled HAL runtime library
├── HAL/inc/            ← alt_types.h, sys/alt_stdio.h, etc.
└── drivers/inc/
    └── altera_avalon_pio_regs.h   ← IOWR_ALTERA_AVALON_PIO_DATA macro
```

## Step 6 — Write and build the C application

The firmware lives in `software/app/main.c`. It is a standalone C program with `main()` as its entry point; the HAL startup code (`crt0.S`) calls `main()` after initialising the runtime.

### 6.1 Understand the source

```c
#include "system.h"                   // LED_PIO_BASE defined here
#include "altera_avalon_pio_regs.h"  // IOWR_ALTERA_AVALON_PIO_DATA

static const uint8_t patterns[] = {
    0x01, 0x03, 0x07, 0x0F, 0x1F, 0x3F, 0x7F, 0xFF,
    0x7F, 0x3F, 0x1F, 0x0F, 0x07, 0x03, 0x01, 0x00,
    0xAA, 0x55, 0xFF, 0x00,
};

int main(void)
{
    uint32_t idx = 0;
    const uint32_t num_patterns = sizeof(patterns) / sizeof(patterns[0]);

    while (1) {
        IOWR_ALTERA_AVALON_PIO_DATA(LED_PIO_BASE, patterns[idx]);
        idx = (idx + 1) % num_patterns;
        delay_ms(200);
    }
}
```

`IOWR_ALTERA_AVALON_PIO_DATA(base, data)` expands to a single 32-bit write to the PIO's data register. The PIO hardware latches the lower 8 bits onto the physical LED pins on the next clock edge.

`LED_PIO_BASE` is not a constant you write by hand — it comes from `system.h`, which is regenerated whenever the hardware changes. This is the principal benefit of the HAL: your C code never needs to know specific addresses.

### 6.2 Build the application

```bash
docker run --rm \
  -v /path/to/cvsoc:/work \
  raetro/quartus:23.1 \
  bash -c "cd /work/04_nios2_led/quartus && make app"
```

The app Makefile compiles with:

```makefile
CC     := nios2-elf-gcc
CFLAGS := -DALT_SINGLE_THREADED=1 -DNIOS2 \
           -I$(BSP_DIR) -I$(BSP_DIR)/HAL/inc -I$(BSP_DIR)/drivers/inc \
           -Os -g -Wall -Wextra \
           -mno-hw-div -mno-hw-mul
LDFLAGS := -T$(BSP_DIR)/linker.x -L$(BSP_DIR) -lhal_bsp -Wl,--gc-sections
```

The `-mno-hw-div -mno-hw-mul` flags tell the compiler to emit software divide and multiply routines instead of hardware instructions — required for the Nios II/e Tiny variant, which has no hardware multiplier.

Expected output:

```
nios2-elf-gcc ... main.c -o nios2_led.elf ...
   text    data     bss     dec     hex
   1404       0       8    1412     584   nios2_led.elf
```

The `.elf` fits well within the 32 KB on-chip RAM.

## Key learnings

These are the five non-obvious problems encountered when building this project from scratch. Each one is a real trap — understanding them will save you hours of debugging.

### 1. Both bus masters must connect to `debug_mem_slave`

The Nios II debug port has its own internal memory slave (`debug_mem_slave`). When the debug port is enabled — the default — Platform Designer validation requires that **both** `data_master` and `instruction_master` are connected to it. Connecting only the data master causes `qsys-generate` to fail with:

```
Error: nios2_system.nios2: Debug port is enabled.
Please connect the instruction_master and data_master to debug_mem_slave
```

**Fix:** Add both connections in your TCL script.

```tcl
add_connection nios2.data_master        nios2.debug_mem_slave
add_connection nios2.instruction_master nios2.debug_mem_slave
```

### 2. IRQ connections use master/slave semantics, not signal direction

In the Avalon interrupt model, the CPU's interrupt controller is the *master* (it receives interrupts); peripherals are *slaves* (they send interrupts). The `add_connection` command is always written `master slave`:

```tcl
# Correct: nios2.irq is the interrupt master (receiver)
add_connection nios2.irq jtag_uart.irq
```

Writing it as `jtag_uart.irq nios2.irq` would either fail validation or wire it backwards.

### 3. Platform Designer systems live in their own VHDL library

When `qsys-generate` produces VHDL, the top-level entity is registered in a separate library named after the system (`nios2_system` in this project). Using a direct entity instantiation (`entity work.nios2_system`) in the top-level VHDL fails at Analysis & Synthesis with:

```
Error (10481): design library "work" does not contain primary unit "nios2_system"
```

**Fix:** Always use a component declaration when instantiating a Platform Designer system from VHDL. The Quartus elaborator resolves the binding from the correct library automatically.

### 4. `qsys-script` does not implement all TCL standard library commands

The `qsys-script` interpreter is a restricted TCL environment. Several standard TCL commands are not available, including `file normalize`. The symptom is a cryptic:

```
Error: sorry, "file normalize" is not implemented yet
```

**Fix:** Avoid TCL file path manipulation in qsys scripts. Instead, structure your Makefile so that `qsys-script` is invoked from the directory where the output file should be saved, and use a plain filename in `save_system`:

```makefile
# Makefile: change into the qsys directory first
cd ../qsys && qsys-script --script=nios2_system.tcl
```

```tcl
# TCL: simple filename, no path manipulation
save_system nios2_system.qsys
```

### 5. `nios2-bsp` breaks inside Docker on WSL2 hosts

The `nios2-bsp` shell wrapper detects the host operating system by checking `uname -r` for the string "microsoft". Because a Docker container on a WSL2 host inherits the WSL2 kernel version string, the check fires inside the container. The script then calls `wslpath` — a WSL-only binary — which is absent inside Docker:

```
/opt/intelFPGA/nios2eds/sdk2/bin//nios2-bsp: line 44: wslpath: command not found
nios2-bsp: Can't mkdir -p
```

The generated BSP `Makefile` has the same problem: it appends `.exe` to the compiler name, producing `nios2-elf-gcc.exe` which does not exist.

**Fix 1:** Call `nios2-bsp-create-settings` directly (the underlying tool the wrapper calls).  
**Fix 2:** Pass `WINDOWS_EXE=` to the generated BSP Makefile to suppress the `.exe` suffix.

```makefile
nios2-bsp-create-settings \
  --sopc  $(SOPCINFO) \
  --type  hal \
  --settings $(BSP_DIR)/settings.bsp \
  --bsp-dir  $(BSP_DIR) \
  --script   /opt/intelFPGA/nios2eds/sdk2/bin/bsp-set-defaults.tcl \
  --cpu-name nios2
$(MAKE) -C $(BSP_DIR) WINDOWS_EXE=
```
